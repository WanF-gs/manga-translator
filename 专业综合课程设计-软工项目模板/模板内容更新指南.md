## 1. 软件调研报告（1-软件调研报告.docx）

### 1.1 项目背景
- **项目名称**：漫画多语言智能翻译与图像合成系统
- **项目定位**：面向全球漫画爱好者的AI驱动一站式翻译工具
- **核心价值**：上传即翻译、一键出成品、多端无缝用、AI持续进化

### 1.2 市场调研
- **目标用户**：个人动漫爱好者、外语学习者、业余内容创作者、漫画出版社(B端)、汉化组/翻译团队
- **竞品分析**：现有工具停留在纯文本翻译阶段，缺乏端到端一站式解决方案
- **差异化优势**：
  - 多模态LLM语境感知翻译（vs 竞品基础NMT）
  - 批量校对工作台（vs 竞品逐页编辑）
  - 轻量团队协作（vs 竞品不支持协作）
  - PWA离线阅读（vs 竞品不支持离线）
  - 有声剧场模式（vs 竞品基础TTS）
  - 动态漫画模式（vs 竞品无此功能）

### 1.3 技术可行性分析
- **前端**：React 18 + Next.js 14 (App Router) + Ant Design 5
- **后端**：Python FastAPI 微服务 + Go (Gin) API网关
- **AI模型**：Qwen2.5-7B翻译 + MangaOCR + LaMa修复 + Real-ESRGAN超分
- **数据库**：PostgreSQL 16 + Redis 7 + MinIO
- **部署**：Docker Compose + GitHub Actions CI/CD

### 1.4 竞品对标分析

| 功能 | 本系统 v3.0 | 竞品A | 竞品B | 差异化优势 |
|------|------------|-------|-------|-----------|
| OCR准确率 | 96% | 90% | 92% | +4%~6% |
| 翻译质量 | 4.3/5 | 3.5/5 | 3.8/5 | 多模态语境感知 |
| 处理速度 | 25秒/页 | 60秒/页 | 45秒/页 | 端云协同加速 |
| 校对效率 | 批量工作台 | 逐页编辑 | 半自动 | 效率提升300% |
| 协作功能 | 轻量协作 | 不支持 | 不支持 | 团队效率提升 |
| 离线能力 | PWA完整离线 | 不支持 | 部分离线 | 移动端体验 |
| API开放 | 5个核心API | 无 | 无 | 生态构建 |
| 有声剧场 | 角色分声+场景音效 | 无 | 基础TTS | 沉浸式体验 |
| 动态漫画 | 运镜+动画+导出 | 无 | 无 | 社交传播引爆点 |

## 3. 系统设计（3-系统设计.docx）

> **参考**：`architecture/系统架构设计说明书.md` (已更新至v2.0)

### 3.1 系统架构概述
- **架构模式**：微服务架构（8个业务服务 + AI Gateway）
- **分层设计**：六层架构（表现层→接入层→业务服务层→AI模型层→异步任务层→数据存储层）
- **部署形态**：B/S网页端 + PWA 2.0离线能力

### 3.2 技术栈选型

| 层级 | 技术选型 | 版本 |
|------|----------|------|
| 前端框架 | React + Next.js (App Router) | 18 + 14 |
| UI组件库 | Ant Design + Ant Design Mobile | 5.x |
| 状态管理 | Zustand | 4.x |
| API网关 | Go (Gin) | 1.9+ |
| 业务服务 | Python FastAPI | 0.110+ |
| 异步任务 | Celery + Redis | 5.x + 7.x |
| 主数据库 | PostgreSQL | 16 |
| 缓存 | Redis | 7.x |
| 对象存储 | MinIO | latest |
| 容器化 | Docker + Docker Compose | - |

### 3.3 数据库设计（20张表）

**核心表**：users, projects, chapters, pages, text_regions
**业务表**：term_entries, style_presets, operation_histories, export_tasks, vocabularies
**系统表**：notifications, translation_cache, user_sessions, reading_progress
**v2.0新增**：characters, translation_quality, feedback, collaboration_locks, comments, api_keys, learning_progress
**v3.0新增**：fonts

### 3.4 接口设计
- RESTful API风格，统一JSON响应格式
- JWT Token认证（Access Token 2h + Refresh Token 7d）
- 4位错误码体系，按模块分段
- 版本化管理 `/api/v1/`
- 17组接口，覆盖全部28大模块

### 3.5 部署架构
- Docker Compose编排17个容器
- 可观测性栈：Prometheus + Grafana + Loki
- GPU支持：AI Gateway内置GPU调度器(gpu_scheduler.py)

## 6. 讨论记录（6-讨论记录.docx）

### 6.1 架构决策讨论
- **讨论主题**：微服务 vs 单体架构
- **结论**：采用微服务架构，AI处理管线各环节独立部署
- **理由**：计算资源需求不同，独立演进，故障隔离

### 6.2 技术选型讨论
- **讨论主题**：API网关语言选择（Go vs Nginx+Lua vs Kong）
- **结论**：使用Go(Gin)
- **理由**：高性能、部署简单、与Python服务解耦

### 6.3 数据库设计讨论
- **讨论主题**：JSONB字段使用策略
- **结论**：OCR结果、翻译结果、样式配置等半结构化数据使用JSONB
- **理由**：PostgreSQL JSONB支持灵活查询，避免过度规范化

### 6.4 部署方案讨论
- **讨论主题**：Docker Compose vs Kubernetes
- **结论**：MVP阶段使用Docker Compose，生产环境迁移K8s
- **理由**：降低MVP阶段运维复杂度

### 6.5 质量保证讨论
- **讨论主题**：擦除质量评估标准
- **结论**：引入SSIM质量评估闭环，低质量自动标记
- **理由**：擦除质量直接影响翻译成品质量，需要量化标准
