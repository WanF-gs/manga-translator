# 漫画翻译渲染回填：当前项目 vs 顶级项目方案对比与改进报告

**日期**: 2026年6月30日  
**查询**: 翻译后文本回填（擦除原文 + 嵌入译文）的当前项目做法与顶级项目做法对比

---

## 执行摘要

当前项目的渲染回填管线整体完整且质量不错——LaMa ONNX 擦除 + 气泡感知椭圆换行 + 双层描边 + 自动描边颜色 + 网点纸保护——这些已经覆盖了漫画翻译的核心需求。但与 manga-image-translator 和 BallonsTranslator 相比，缺失了三项对最终成品质量影响巨大的能力：(1) 原文样式自动匹配（字体颜色、描边、阴影），这是 BallonsTranslator 的核心差异化功能；(2) CJK 禁则处理（标点不出现在行首/行尾），中文日文排版没有这个就很业余；(3) 基于气泡精确掩码的 balloon_fill 布局模式，当前用椭圆近似代替精确气泡形状，精度有差距。此外，擦除模块用了通用 LaMa 而非漫画数据微调版，对网点纸和速度线的修复效果不如 BallonsTranslator 的微调版。

---

## 一、当前项目渲染回填管线概览

### 1.1 擦除模块（Inpainting）

**三级降级链**：LaMa ONNX (186MB) → OpenCV TELEA → 边缘感知颜色填充

**高光设计**：
- **双重阈值文字掩码**：OTSU + 自适应高斯阈值取交集，避免误擦背景
- **网点纸 FFT 频谱检测**：通过 2D FFT 检测周期性峰值，判断是否为网点纸背景
- **LBP + GLCM 背景分类器**：5 种背景类型（网点纸、速度线、渐变、纯色、复杂纹理），据此调整擦除策略
- **擦除质量评估**：SSIM(50%) + GMSD(25%) + BRISQUE(25%)，分背景类型设不同通过阈值，低于 60 分自动标记 needs_repair

**问题所在**：使用的是通用 LaMa ONNX 模型，未针对漫画数据微调。网点纸和速度线的修复质量不稳定，容易产生模糊或纹理断裂。

### 1.2 排版引擎

**核心能力**：
- 二分搜索最优字号（6px ~ 48px，按区域类型约束范围）
- 气泡感知椭圆换行：用椭圆方程计算每行可用宽度，而非矩形
- 自动缩字号：字号逐步降至初始的 70%，仍溢出则截断加 "..."
- 双层描边：外层粗描边（八方向+对角），内层细描边（四方向），主文字在最上层
- 自动描边颜色：深色文字（亮度 < 128）→ 白色描边，浅色文字 → 黑色描边
- 最大内接矩形搜索：对不规则轮廓使用滑动窗口 + pointPolygonTest

**问题所在**：
- CJK 换行是逐字换行，没有禁则处理——标点可能出现在行首
- 用椭圆近似代替精确气泡轮廓，圆形气泡 OK，不规则气泡有偏差
- 没有原文样式自动匹配——翻译文字的风格（颜色、描边、阴影）不会自动跟随原文

### 1.3 渲染实现

- 竖排文字逐字符纵向排列，列从右到左
- 坐标校准：API 尺寸与实际图片尺寸不一致时自动缩放
- 前端 Rich 编辑器支持手动调整样式、笔刷修补、撤销/重做

## 二、顶级项目做法对照

### 2.1 Inpainting 对比

| 维度 | 当前项目 | manga-image-translator | BallonsTranslator |
|------|----------|------------------------|-------------------|
| 主力模型 | 通用 LaMa ONNX | LaMa / LaMa-MPE / LDM Diffusion | **漫画微调 LaMa** + PatchMatch |
| 大遮罩质量 | 中等（网点纸区域有模糊） | LDM 下 33.5dB PSNR | 微调后网点纸修复显著优于通用版 |
| 速度 | 快（ONNX 推理） | LaMa 快 / LDM 慢 | PatchMatch 最快 / 微调 LaMa 中等 |
| 漫画专用性 | 有网点纸检测但修复模型未适配 | LDM 扩散模型生成能力强但慢 | **漫画数据微调的 LaMa 是最佳平衡点** |

**关键差异**：BallonsTranslator 在漫画数据上微调了 LaMa 模型，对网点纸（screentone）、速度线（speedline）、渐变背景的修复效果显著优于通用 LaMa。当前项目的通用 LaMa 面对网点纸时容易产生模糊块或纹理断裂，虽然做了后处理的网点纸感知混合，但效果有限。

### 2.2 Typesetting 对比

| 维度 | 当前项目 | manga-image-translator | BallonsTranslator |
|------|----------|------------------------|-------------------|
| 字号搜索 | 二分搜索（区域类型约束） | 二分搜索（balloon_fill 气泡掩码内） | 自动 + 全局设置 |
| 气泡适配 | **椭圆近似** | **精确气泡掩码** (balloon_fill) | 自动估算 |
| CJK 换行 | 逐字换行 | **禁则处理** + 竖排括号压缩 | pkuseg 中文分词断句 |
| 样式匹配 | ❌ 无自动匹配 | 基础（前景/背景色、方向） | **完整 Auto-Style**（颜色、描边、阴影、轮廓、角度、字体识别） |
| 渲染后端 | PIL + OpenCV | QPainter + FreeType | QTextDocument |
| 透视变换 | ❌ 无 | **Homography + warpPerspective** | 旋转支持 |
| 竖排优化 | 基本逐字符 | **CJK 兼容形式**（︵/︶）、长音符旋转 | 标点符号集分类旋转 |

### 2.3 三种布局模式（manga-image-translator）

manga-image-translator 提供三种可切换的布局模式，这是当前项目最值得借鉴的设计：

1. **strict**：保持原始文本行数不变，先缩小字号适配、再放大到初始字号。适合"译者想控制排版"的场景。

2. **smart_scaling**：自动确定最佳换行点，动态字号根据区域宽高比扩展文本框。日常翻译的默认推荐模式。

3. **balloon_fill**：在气泡的精确像素掩码内二分搜索最大字号。文本必须完全位于气泡遮罩内。这是质量最高但也最慢的模式——需要先有气泡级别的精确掩码。

当前项目本质上只有一种模式，介于 smart_scaling 和简化版 balloon_fill（用椭圆代替真实掩码）之间。

### 2.4 CJK 禁则处理（manga-image-translator）

这是当前项目缺失的最直接影响排版质量的特性。禁则处理是一套标点符号位置规则：

- **行首禁止**：不允许出现在行首的标点（如 。、，》）→ 上一行多带一个字
- **行尾禁止**：不允许出现在行尾的标点（如 （《「）→ 下一行多带一个字
- **竖排括号压缩**：竖排时括号宽度压缩为半角（0.5em），节省列宽
- **竖排 CJK 兼容形式**：`(` → `︵`、`)` → `︶`，使用 Unicode 竖排专用字符

没有禁则处理的 CJK 排版看起来会非常业余，尤其在漫画这种视觉出版物中尤为明显。

### 2.5 原文样式自动匹配（BallonsTranslator）

这是 BallonsTranslator 区别于其他方案的核心能力。系统会自动检测原文的：
- 字体颜色（前景色）
- 描边颜色和粗细
- 阴影效果
- 文字轮廓
- 倾斜角度
- 水平/垂直方向
- 对齐方式
- 可选字体识别（YuzuMarker.FontDetection）

然后自动将这些样式应用到翻译文本上。这极大地减少了对同一幅图多次调样式的工作量。

## 三、当前项目 vs 顶级方案：差距分析

### 差距 1：CJK 禁则处理（影响最大，实现成本最低）

**问题**：当前 CJK 换行是逐字拆分，不考虑标点位置合法性。结果：`。` `、` `）` 等标点可能孤零零出现在行首。

**影响**：中文/日文读者对禁则极为敏感，违反禁则的排版一眼就能看出不专业。

**实现难度**：低。在现有 `text_layout.py` 的换行逻辑中插入一个标点检查步骤即可。约 50-80 行代码。

### 差距 2：原文样式自动匹配

**问题**：翻译文字使用了固定的样式（白色描边/黑色描边自动选择），不会跟随原文风格。

**影响**：如果原文是红色文字加黄色描边（常见于呐喊/强调），翻译后变成黑字白描边，风格断裂明显。

**实现难度**：中。需要在 OCR 阶段或擦除前从原文区域提取颜色直方图和边缘分析；约 150-250 行代码。

### 差距 3：精确气泡掩码换行（balloon_fill）

**问题**：用椭圆方程近似气泡可用宽度，对圆形气泡 OK，但对不规则形状（如多边形气泡、带有尖尾的气泡）精度不够。

**影响**：文字可能延伸进气泡的尖尾区域，或在不规则边界处超出气泡。

**实现难度**：中高。需要气泡级别的精确分割掩码（当前检测阶段未提供），然后对每个行 Y 坐标查询该位置的气泡水平跨度。约 200-300 行代码，且依赖检测阶段提供气泡掩码。

### 差距 4：漫画数据微调 LaMa

**问题**：通用 LaMa 对网点纸和速度线的修复质量不足，会产生模糊或纹理不连续。

**影响**：擦除质量是翻译成品的第一印象。擦除不好，后面排版再精致也白搭。

**实现难度**：高。需要漫画擦除标注数据集 + GPU 微调训练。不适合短期解决。

## 四、具体改进建议（按优先级排序）

### 优先级 P0：添加 CJK 禁则处理

**修改文件**：`text_layout.py` 和 `renderer.py`

**做法**：在现有的 CJK 逐字换行逻辑之后，加一个后处理步骤——扫描每一行，如果行首是禁则标点（`。，、》）！？…—~・` 等），则从上一行末尾"借"一个字过来；如果行尾是行尾禁则标点（`（《「『【` 等），则从下一行开头"推"一个字过去。

manga-image-translator 的 `auto_linebreak.py` 中有完整的禁则标点列表和转移逻辑，可以直接参照实现。这是当前项目排版质量提升最快、性价比最高的改动。

### 优先级 P1：原文颜色自动匹配

**修改文件**：`render_service.py` 或新的 `style_extractor.py`

**做法**：在擦除之前，对每个文字区域做以下分析：
1. 用 K-Means (k=2) 对 ROI 像素聚类，分离文字色和背景色
2. 取暗聚类中心作为文字颜色（漫画文字通常深于气泡背景）
3. 在文字周围检测描边像素（灰度梯度突变且亮度介于文字与背景之间）
4. 如果有描边，取描边像素的聚类中心作为描边颜色

BallonsTranslator 的做法是通过 CNN 分析像素级特征，但用传统 CV 方法（颜色聚类 + 梯度分析）在漫画场景下也能达到 80% 以上的准确率，实现成本低得多。

### 优先级 P2：竖排 CJK 兼容形式

**修改文件**：`renderer.py`

**做法**：在竖排渲染时，将标准括号替换为竖排专用 Unicode 字符：
- `(` → `︵` (U+FE35)
- `)` → `︶` (U+FE36)
- `「` → `﹁` (U+FE41) 等

同时将竖排中的英文数字用当前项目已有的水平块逻辑处理（已有基础），并为括号字符使用半角宽度而非全角。

### 优先级 P3：精确气泡掩码换行

**修改文件**：`text_layout.py`

**前提**：需要检测阶段提供气泡级别掩码（当前已有 `boundary_polygon` 字段，可在此基础上构建精确掩码）。

**做法**：将 `_get_bubble_available_width` 中的椭圆方程计算替换为：对该 Y 坐标从掩码左边界扫描到右边界，取连续非零段的最大宽度。

## 结论

当前项目的渲染回填管线在擦除质量评估、网点纸检测、双层描边等方面已经做得比很多开源项目更精细。与顶级方案的差距主要集中在三点：CJK 禁则处理、原文样式自动匹配、和精确气泡掩码换行。这三项的改进成本从低到中等，且都能在现有架构内实现，不需要替换模型或重构管线。其中 CJK 禁则处理的性价比最高——几十行代码就能消除中文排版中最明显的业余痕迹，建议优先实现。
